Demo Plugin ships with a PHPUnit setup that runs exclusively inside the `wp-env` test container. Running tests outside of `wp-env` is not supported â€” the bootstrap will exit immediately if `WP_TESTS_DIR` is not set, which only happens inside the container.

The container provides a real WordPress database, the full WordPress function library, PHPUnit, and the `yoast/phpunit-polyfills` library required by the WordPress test bootstrap. The bootstrap lives in `tests/bootstrap.php`, configuration in `phpunit.xml.dist`, and test classes go in `tests/php`.

## Write your first test

<Steps>
1. Start the test environment.

```bash title="Terminal"
npm run env:start
```

2. Add a test class in `tests/php`. An `ExampleTest.php` is already provided as a starting point.

```php title="tests/php/MyFeatureTest.php"
<?php

class MyFeatureTest extends WP_UnitTestCase {
	public function test_something(): void {
		$this->assertTrue( true );
	}
}
```

3. Run tests.

```bash title="Terminal"
npm run test:php
```
</Steps>

:::note[Why `tests-cli` and not `cli`?]
`wp-env` exposes two containers. `cli` is the development container sharing the main database. `tests-cli` is the dedicated test container with its own isolated database and `WP_TESTS_DIR` pre-configured to the bundled WordPress test framework. `test:php` uses `tests-cli` directly â€” no Composer wrapper â€” so the PHPUnit version in the container is always matched to the bundled WP test files.
:::

## Common test patterns

### WordPress data with factories

:::note[Automatic database rollback]
`WP_UnitTestCase` wraps each test in a transaction and rolls it back automatically when the test finishes. You never need to manually clean up posts, users, or meta â€” each test starts with a clean slate.
:::

```php title="tests/php/PostMetaBehaviorTest.php"
<?php

class PostMetaBehaviorTest extends WP_UnitTestCase {
	public function test_can_store_and_read_post_meta(): void {
		$post_id = self::factory()->post->create();
		update_post_meta( $post_id, '_demo_flag', 'yes' );

		$this->assertSame( 'yes', get_post_meta( $post_id, '_demo_flag', true ) );
	}
}
```

### Hooks and filters

```php title="tests/php/FilterRegistrationTest.php"
<?php

class FilterRegistrationTest extends WP_UnitTestCase {
	public function test_demo_filter_modifies_value(): void {
		add_filter(
			'demo_plugin/example_value',
			static fn( string $value ): string => $value . '-modified'
		);

		$result = apply_filters( 'demo_plugin/example_value', 'before' );
		$this->assertSame( 'before-modified', $result );
	}
}
```

## Plugin dependencies for testing

Some tests require other plugins to be present and active in the test environment.

### Free plugins from WordPress.org

Use WP-CLI's `plugin install` command inside the `tests-cli` container. Extend the existing `lifecycleScripts.afterStart` command in `.wp-env.json` so it runs automatically on `env:start`:

```json title=".wp-env.json"
{
	"lifecycleScripts": {
		"afterStart": "<existing_commands> && wp-env run tests-cli wp plugin install advanced-custom-fields --activate"
	}
}
```

The plugin slug is the same as on wordpress.org/plugins. You can also pin a specific version:

```bash
wp-env run tests-cli wp plugin install advanced-custom-fields --version=6.3.0 --activate
```

### Premium or private plugins

Private and premium plugins are managed through a dedicated `tests/composer.json`. Composer installs them into `tests/setup/plugins/`, which is mapped into the wp-env WordPress installation before the environment starts.

Run `npm run env:install-test-deps` before `npm run env:start` to ensure the plugins are present when the environment starts. On CI this step runs explicitly before wp-env.

<Steps>
1. Add the private Composer repository and package to `tests/composer.json`. Use an explicit installer path keyed to the package name so the install directory is predictable.

   ```json title="tests/composer.json"
   {
     "repositories": [
       {
         "type": "composer",
         "url": "https://composer.my-premium-plugin.com"
       }
     ],
     "require": {
       "vendor/my-premium-plugin": "*"
     },
     "extra": {
       "installer-paths": {
         "setup/plugins/my-premium-plugin/": ["vendor/my-premium-plugin"]
       }
     },
     "config": {
       "allow-plugins": {
         "composer/installers": true
       }
     }
   }
   ```

   Add one `installer-paths` entry per package. The directory name you choose here must match the mapping in `.wp-env.json` in the next step.

2. Map each installed plugin directory into wp-env in `.wp-env.json`.

   ```json title=".wp-env.json"
   {
     "mappings": {
       "wp-content/plugins/demo-plugin": ".",
       "wp-content/plugins/my-premium-plugin": "tests/setup/plugins/my-premium-plugin"
     }
   }
   ```

   wp-env requires one explicit mapping per plugin. Pointing the mapping at `tests/setup/plugins` as a whole directory is not supported.

3. Create `tests/auth.json` locally with credentials for the private repository.

   ```json title="tests/auth.json"
   {
     "http-basic": {
       "composer.my-premium-plugin.com": {
         "username": "your-license-email",
         "password": "your-license-key"
       }
     }
   }
   ```

   :::caution
   `tests/auth.json` is listed in `.gitignore` and must never be committed.
   :::

4. Start the environment. Composer installs the plugins before wp-env launches.

   ```bash title="Terminal"
   npm run env:start
   ```

5. For CI, add a `COMPOSER_AUTH` secret in GitHub Actions with the same JSON content as `auth.json`. The `test-application` workflow writes it to `tests/auth.json` before `env:start` runs.

   ```json title="GitHub Actions secret: COMPOSER_AUTH"
   {
     "http-basic": {
       "composer.my-premium-plugin.com": {
         "username": "your-license-email",
         "password": "your-license-key"
       }
     }
   }
   ```

   :::note
   If `COMPOSER_AUTH` is not set, the write step is skipped and only the packages in `tests/composer.json` that require no authentication are installed.
   :::
</Steps>

## Configuration notes

**Namespaces in test files** â€” `WP_UnitTestCase` is a global class so test classes conventionally use no namespace. If you prefer a namespace, add a `Tests\` autoload entry in `composer.json`.

**Local `phpunit.xml`** â€” `phpunit.xml.dist` is committed as the shared default. Create a local `phpunit.xml` to override settings for your environment (it is already listed in `.gitignore`).


## CI

The `test-application` job in `.github/workflows/test-analyse.yml` runs the full test suite on every push via wp-env.
